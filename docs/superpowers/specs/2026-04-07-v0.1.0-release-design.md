# ks-types v0.1.0 发布设计

## 范围

ks-types 代码补全 + v0.1.0 tag 发布 + ks-admin/ks-hub 切换到正式版本（删除 replace 指令）。

## 实施策略

串行推进：补全改进 → 全量测试 → 提交推送 → 打 tag → ks-admin 切换 → ks-hub 切换。每一步有明确验证点。

---

## 第一部分：ks-types 4 项改进

### 1. DefaultPermissionRegistry() 直接单元测试

**文件**：`permissions_test.go`

新增 `TestDefaultPermissionRegistry`，验证：
- 返回的 registry 非 nil
- 预置 4 个维度（network、llm、filesystem、user_context）全部存在
- 每个维度：Levels 非空、RiskWeight > 0
- 用一个 valid permission 调 `Validate()` 确认无 error

### 2. BizError HTTP 状态码映射

**文件**：`errors.go` 新增方法

```go
// HTTPStatus 根据错误码前缀推导 HTTP 状态码
func (e *BizError) HTTPStatus() int
```

映射规则（基于现有错误码分段）：
- `400xx` → 400
- `401xx` → 401
- `403xx` → 403
- `404xx` → 404
- `409xx` → 409
- `500xx` → 500
- `503xx` → 503
- 其他 → 500（兜底）

**测试**：`errors_test.go` 新增 `TestBizErrorHTTPStatus`，覆盖所有现有错误码 + 未知码兜底。

### 3. PermissionRegistry 线程安全

**文件**：`permissions.go`

- 给 `PermissionRegistry` 加 `sync.RWMutex`
- `Register()` 用写锁
- `Validate()` 和 `HighRiskPermissions()` 用读锁
- 不改导出 API，纯内部改动

**测试**：`permissions_test.go` 新增 `TestPermissionRegistryConcurrency`，用 goroutine 并发读写验证无 race。

### 4. 恶意 JWT token 边界测试

**文件**：`instance_claims_test.go`、`developer_claims_test.go`

新增测试用例：
- 空字符串 token
- 随机垃圾字符串
- 截断的合法 token（只保留 header 部分）
- 被篡改的 payload（base64 decode → 改内容 → re-encode）

所有用例均应返回 error，不 panic。

## 第二部分：Tag 发布流程

1. 全量测试：`go test ./...`
2. 竞态检测：`go test -race ./...`
3. 静态检查：`go vet ./...`
4. 提交改进代码：一个 commit 包含全部 4 项改进
5. 推送 master：`git push origin master`
6. 打 tag：`git tag v0.1.0`
7. 推送 tag：`git push origin v0.1.0`
8. 验证发布：`go list -m github.com/wuhanyuhan/ks-types@v0.1.0`

---

## 第三部分：ks-admin / ks-hub 切换正式版本

### 每个项目的步骤（先 ks-admin，后 ks-hub）

1. 删除 `go.mod` 中的 `replace github.com/wuhanyuhan/ks-types => ../ks-types`
2. `go get github.com/wuhanyuhan/ks-types@v0.1.0`
3. `go mod tidy`
4. `go test ./...` 确认无破坏
5. 提交：`chore: 切换 ks-types 到 v0.1.0 正式版本`
6. `git push origin master`

### 失败回滚策略

- 测试失败时不提交、不推送
- 分析原因，如果是 ks-types 问题则回到 ks-types 修复 → 发 `v0.1.1` patch → 重试
- 本次 4 项改进不涉及破坏性 API 变更，回归风险低

---

## 版本选择说明

选择 `v0.1.0` 而非 `v1.0.0`，原因：
- API 尚未经过多服务大规模验证
- 后续 keystone 整合时可能需要扩展错误码或微调结构
- v0.x 在 Go 语义上不承诺向后兼容，给了调整空间
